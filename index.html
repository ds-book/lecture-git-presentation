❓ Что такое Git ?

Система контроля версий (Version Control System, VCS) — это система, записывающая изменения в файл или набор файлов в течение времени и позволяющая вернуться позже к определённой версии.

Git (произносится «гит») — распределённая система управления версиями.

Проект был создан Линусом Торвальдсом для управления разработкой ядра Linux, первая версия выпущена 7 апреля 2005 года.

Отслеживает изменения

Организует взаимодействие нескольких разработчиков

📦 Репозиторий: Контейнер проекта

Локальный репозиторий (local repository) хранится на вашем компьютере, где вы можете работать над локальной версией проекта.

Удаленный репозиторий (remote repository) хранится вне вашей локальной системы, обычно на сервере. Это позволяет разработчикам делиться кодом.

Создание локального репозитория: git init

После этой команды создается папка .git в которой будут храниться служебные файлы и файл .gitignore

🚫 .gitignore: Настройка игнорирования файлов

🗂️ Репозиторий = Папка с проектом
├── 📄 Working directory — текущие изменения
├── 📋 .gitignore — файлы, которые не нужно отслеживать
└── 🗄️ .git — архив всех версий (история)

🗄️ .gitattributes: Хранение больших файлов

Иногда возникает необходимость хранения больших файлов в репозитории.

Это могут быть наборы данных, модели машинного обучения, инструкции в PDF и т.д.

Подразумевается, что такие файлы нужно сохранить в репозитории, но их не нужно отследивать.

В этом случае необходимо использовать расширение Git Large File Storage (LFS) https://git-lfs.com.

git lfs install                 # Установить расширение для репозитория
git lfs track "*.pdf"           # Добавляем файл в хранилище
git add .gitattributes          # Добавляем файл с атрибутами добавлен в индекс

📊 Состояния файлов в репозитории

Состояние показывает: актуален ли репозиторий, есть ли там новые/измененные/удаленные файлы или неотслеживаемые (те, которые еще не добавлены в репозиторий) файлы.

git status

Возможные состояния:

  • Не индексирован (Untracked)
  • Не изменено (Unmodified)
  • Изменено (Modified)
  • Подготовлено (Staged).

📋 Индекс (staging area)

Индекс — области подготовленных файлов - промежуточное место между вашим прошлым коммитом и следующим. Вы можете добавлять или удалять файлы из индекса.

Вы можете добавлять файлы в индекс (по одному и сразу все)

git add filename.txt, file2.txt    # Добавить файлы в индекс
git add .                          # Добавить ВСЕ файлы в индекс
git reset filename.txt             # Удалить файл из индекса
git restore filename.txt           # Сбросить изменения файлов
git diff                           # Показать изменения в файле

👁️ Отображение изменений в файле

💾 Фиксация изменений (commit)

Коммит – это снимок кода проекта в определенный момент времени, который сохраняется в истории коммитов репозитория. Обозначаются хешем (уникальная последовательность символов)

Коммит делается после добавления всех отслеживаемых файлов в индекс с помощью команды git add

git commit -m "Сообщение коммита". # Создать коммит
git log                            # Посмотреть историю коммитов
git checkout <хеш коммита>         # Вернуться на состояние какого либо коммита
git checkout master                # Вернуться к последнему коммиту
git commit --amend -m "hotfix"     # Исправить последний коммит
git revert <хеш коммита>           # Отменить изменения коммита
git revert HEAD                    # Отменить изменения последнего коммита

🏷️ Теги (tags)

Теги – это метки, предназначенные для обозначения важных коммитов или этапов завершения работы.

git tag ver-2.5                    # обычный тег (облегченный)
git tag -a ver-2.5 <хеш коммита>   # тег на выбранный коммит
git tag -a ver-2.5 -m "beta 2.5"   # тег с сообщением (аннотированный)
git tag                            # показать все теги
git tag -l *-rc*                   # показать теги, которые содержат -rc
git checkout <тег>                 # перейти на коммит с тегом
git tag -a -f <тег> <хеш коммита>  # переопределить тег
git tag -d <тег>                   # удалить тег

🎯 Рабочий цикл

git init                          # Создать репозиторий
git status                        # Проверить состояние (самая частая!)
git add .                         # Добавление файл в индекс
git commit -m "done"              # Фиксация изменений

📝 Аналогия: Подготовка статьи к публикации

1️⃣ Написали черновик → файл изменён (Modified)

2️⃣ Проверяем статус файлов → git status (Untracked)

2️⃣ Выбрали, что отправить → git add (Staged)

3️⃣ Отправили в журнал → git commit (Committed)

4️⃣ Статья опубликована → можно ссылаться на версию

🌿 Ветвление: Эксперименты без риска

🌳 Ветви (branches)

Ветвь или ветка – одна из версий репозитория, которая отличается от основного рабочего проекта или от других версий.

По умолчанию в репозитории всегда есть “основная” ветка master / main.

Это позволяет нам добавлять новые (экспериментальные, незавершенные и потенциально опасные) функции в отдельных ветках, не трогая официальную стабильную версию кода, которая обычно хранится в ветке мастер (master).

🔀 Слияния веток (merge)

Ветви можно объединять, если хотите внедрить сделанные в одной из них изменения кода в другую.

![]

📋 Порядок действий при слиянии

Перед объединением веток нужно переключиться в ту ветку, в которую будут переноситься изменения.

Например, если сливаем изменения из ветки develop с веткой main, то

1️⃣ Переключаемся в ветку main:

git checkout main

2️⃣ Сливаем develop с main:

git merge develop

После слияния ветка не удаляется.

🛠️ Команды для работы с ветвями

git log --oneline --graph         # история коммитов с ветками
git branch <название новой ветви> # создание ветки
git checkout -b <название>        # создание ветки и сразу переход на нее
git checkout <название ветви>     # переход между ветками
git merge <branch>                # объединить ветку branch с текущей
git branch -d <name>              # удалить ветку

🔬 Пример: Параллельные исследования

🌳 main (стабильная версия)
│
├─ 🧪 feature/new-biomarker
│  ├─ Тестируем новый маркер
│  └─ Не сработало? → удаляем ветку
│
├─ 🐛 fix/statistical-error  
│  ├─ Исправляем баг изолированно
│  └─ После проверки → мержим в main
│
└─ 📊 experiment/alt-model
   ├─ Пробуем альтернативную модель
   └─ Сравниваем, выбираем лучший
# Создаём ветку для эксперимента
git checkout -b experiment/alt-model  

# Вносим изменения только в этой ветке
# 📝 Модифицируем normalization.R

# Проверяем, что main не затронут
git checkout main → старая версия
  
# Если успешно — мержим
git merge experiment/alt-model
  
# Если нет — удаляем
git branch -D experiment/alt-model

🛠️ Команды для работы с ветками

git branch                        # Список веток
git checkout -b <name>            # Создать и перейти
git checkout <name>               # Переключиться
git merge <branch>                # Объединить с текущей
git branch -d <name>              # Удалить ветку

⚠️ Конфликты при слиянии

При слиянии ветвей может возникнуть ситуация конфликта – когда изменения затронули один и тот же файл в одном месте и нельзя автоматически определить какие правки актуальны.

$ git merge feature
Auto-merging print_array.js
CONFLICT (content): Merge conflict in print_array.js
Automatic merge failed; fix conflicts and then commit the result.
$ git add .
$ git commit -m "Array printing conflict resolved."

☁️ Удаленный репозиторий

Удаленный репозиторий – это версии вашего проекта, сохраненные на удаленном сервере. Доступ к репозиторию на таком сервере может осуществляться по интернету или по локальной сети.

Это полноценный репозиторий, ничем не отличающийся от локального. У удаленного репозитория есть собственные ветки, собственный указатель HEAD, своя история коммитов и так далее.

Обычно обозначается как origin.

🔗 Подключение удаленного репозитория

Если мы подключим удаленный репозиторий к своему локальному, то у нас появятся копии всех ссылочных объектов удаленного репозитория. То есть, например, у удаленного репозитория есть ветка main, а у нас будет копия этой ветки – origin/main.

# Добавление удаленного репозитория к локальному
git remote add <название удаленного репозитория> <ссылка на удаленный репозиторий>

# Показать текущий удаленный репозиторий
git remote show <название удаленного репозитория>

# Удалить привязку удаленного репозитория к локальному
git remote remove <название удаленного репозитория>

🛠️ Работа с удаленным репозиторием

🔄 Рекомендуемый порядок:

  1. Перед началом:
    git pull origin main

  2. После изменений:
    git add . git commit -m "msg"

  3. Отправка:
    git push origin main

  4. При конфликтах:
    Решить → git add → git commit
